Loop Engineering 还是 Harness Engineering？被过度包装的 /loop 命令

!封面

这两天，我的信息流被"Loop Engineering"占领了。

大量 AI 博主在推送同一个叙事：AI 工程方法已经走完了 Prompt Engineering → Context Engineering → Harness Engineering 的进化路线，现在正式进入 Loop Engineering 时代。文章一篇比一篇恢宏，好像我们正在见证范式革命。

但我越读越觉得不对劲。

我每天都在用 Claude Code。/loop 这个命令我太熟了。它就是 Claude Code 触发循环定时任务的一个内置指令。你给它一个 prompt 和一个时间间隔，它每隔一段时间自动跑一次。

仅此而已。

它是个有用的功能。但把它包装成一个独立的"Engineering"时代，是典型的术语通胀。

/loop 到底做了什么

先看清楚 /loop 的实质。

当你在 Claude Code 里输入 /loop 5m "检查部署状态"，系统每隔 5 分钟自动执行一遍"检查部署状态"。省去了手动反复输入的麻烦，没毛病。

但关键在于，每次执行的时候，整个运行流程长什么样？

权限检查。这个任务有没有被授权执行？工具路由。这个任务要调用哪些工具，走什么优先级？输出治理。结果怎么返回，怎么格式化？错误恢复。执行失败了怎么处理，重试几次？

答案很明确：全部由 Harness 体系控制。Loop 只做了一件事——在指定的时间点，发出"该跑了"这个信号。它不定义跑什么、怎么跑、跑完怎么办。这些问题的答案从来都在 Harness 那里。

打个比方：给一辆车加装定速巡航，你不会因此宣布汽车工业进入了"巡航工程"时代。定速巡航只是对油门的一个定时控制信号。整辆车的动力系统、制动系统、安全系统，仍然是汽车工程的产物。Loop 之于 Harness，就是定速巡航之于汽车。

为什么这不是一个新范式

一个方法论要成为独立范式，至少得满足一个条件：它定义了此前不存在的新控制结构，改变了我们解决问题的基本方式。

Loop 两点都不及格。

第一，Loop 没有引入新控制结构。在 Claude Code 的架构里，/loop 本质上是 Agent Loop 的特化。Agent Loop 本身就包含"接收输入 → 执行任务 → 输出结果 → 等待下一个输入"这个循环。/loop 只改了一件事：把"等待下一个输入"从"等用户打字"换成了"等定时器响"。循环体的内部结构纹丝未动。

第二，整个方法论的构建基础仍然是 Harness。能力边界的定义、安全策略的执行、工具链的编排、输出质量的治理——这些决定 Agent 行为质量的东西，全部来自 Harness。Loop 没有给其中任何一个提供新方案。你不能把"给 Harness 加了一个 cron 表达式"说成"开创了一个新工程时代"。

第三，范式改变的是解题方式，不是执行频率。Prompt Engineering 改变了我们跟模型的交互方式——从写代码变成了写提示。Context Engineering 改变了我们管理信息的方式——从单轮对话变成了结构化上下文注入。Harness Engineering 改变了我们设计 Agent 的方式——从一次性模型调用变成了有约束有治理的运行体系。这些都是解题方式的变化。Loop Engineering 改变了什么？它让 Harness 多跑了几次。解题方式原封不动。

认真对待反面意见

有人说我太苛刻了。Loop 确实产生了质变——它让 AI Agent 从"人在回路中等你下指令"变成了"自主定时运行"。这是从工具到代理的关键一步。

我不否认运行模式变了。但运行模式的变化和工程方法的变化，是两回事。

Agent 从被动变主动，确实是质变。但这个变化发生在 Harness 的框架内部。Harness 本身就设计了"Agent 可以在无人干预下自主运行"这个能力，Loop 只是激活了其中的一种使用方式。Harness 设计了发动机，Loop 是点火开关。点火确实让发动机转起来了，但你不会说"点火工程"取代了"发动机工程"。

退一步想：如果 Loop 真是一种独立的 Engineering，那我们应该能用 Loop 的方法论去指导和评价 Agent 系统的设计。实际上呢？你评价一个 Loop 任务的执行质量时，评估的仍然是它的 Harness 设计——权限设得合不合理、工具链配置得好不好、错误处理够不够健壮。Loop 本身没有提供任何评估维度。

怎么识别"新词制造"

这不是我第一次看到"X Engineering"被过度包装。这个领域的术语通胀速度，可能比实际工程进步快一个数量级。

我给自己总结了一个简单框架，用来区分真范式和命名泡沫。三个问题：

它是否定义了新的控制结构？ 真范式会引入此前不存在的抽象层或控制机制。Harness 引入了运行时的能力边界和治理层，这是新的。Loop 没有。

它是否改变了问题求解的基本方式？ Prompt Engineering 让你从"写代码求解"变成了"写提示求解"。方式变了。Loop 让你从"手动触发求解"变成了"定时触发求解"，求解本身没变。

去掉这个词，体系是否仍然完整？ 把"Harness Engineering"从 Agent 设计的讨论中拿掉，整个控制结构、安全模型、治理框架的概念就会缺失。但把"Loop Engineering"拿掉，你只需要说一句"Harness 支持定时任务"。体系毫发无损。

三道题，Loop Engineering 全部零分。

我不是反对创造新词。一个领域要往前走，需要新概念来标记新进展。Prompt Engineering、Context Engineering、Harness Engineering——这些词能立住，是因为它们各自锚定了一组新的工程实践和设计原则。

但我反对用词汇创新替代实质创新。"Loop Engineering"让人警惕，不是因为它说了什么，而是因为它没说什么。没定义新原则，没引入新结构，没改变解题方式。它只是把已有体系中的一个配置项，包装成了一个时代。

下次有人在你的时间线上宣布"X Engineering"时代来临，先别急着转发。问问 X 到底做了什么。如果答案只是"它让某个已有的东西多跑了几次"——那不是工程革命，是营销革命。
