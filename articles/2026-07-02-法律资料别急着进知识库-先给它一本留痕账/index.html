法律资料别急着进知识库，先给它一本留痕账

!封面

很多法律人搭知识库，第一反应是“我有多少资料”。

合同模板、裁判文书、法规汇编、项目备忘、培训 PPT、检索报告，越多越安心。资料一多，再接一个 AI 问答工具，好像就离“智能法律知识库”不远了。

我现在越来越觉得，这个顺序反了。

法律资料进知识库之前，先要回答几个朴素问题：这份资料从哪里来？谁改过？删掉了什么？脱敏有没有做？能不能回到旧版本？哪些内容只能自己看，哪些可以给团队看，哪些可以对外展示？

这些问题答不清，知识库越大，风险越大。

所以在四明山法师 AI 夜校 Day 6，我把作业设计成一个“最小法律数据闭环”：识别、采集、加工、治理、应用、发布判断。听起来像数据课，其实是在训练法律资料进入 AI 工作流前的基本卫生。

其中最容易被低估的一步，是 Git 留痕。

Git 不是程序员玩具，是资料的变更账本

先把话说直：法律人不需要把自己训练成程序员。

但法律人需要理解版本。

一份合同审查清单，第一版有 30 个风险点，第二版删掉了 8 个，第三版把“解除权”改成“终止权”，第四版又加了数据合规条款。三个月后团队复盘，谁还能说清楚为什么改？

如果只靠文件名，结果通常是这样：

“最终版”

“最终版-修改”

“最终版-客户确认”

“最终版-客户确认-石总改”

“最终版-客户确认-石总改-真的最终”

这不是玩笑。很多争议、返工和知识资产流失，就发生在这些文件名里。

Git 的价值不在于神秘命令，而在于它强迫你每次变化都留下一句话：我改了什么，为什么改。

对法律资料来说，这句话就是变更账本。

六步闭环，比一座大知识库更重要

我给学员的最低要求不是“搭一个很大的库”，而是把一组真实材料走完六步。

第一步，识别。

不要一上来就全盘导入。先选最近一个月还会再次使用的材料。比如一类合同模板、一批案件研究资料、某个项目的尽调文件、某个专题法规与案例。

资料值得沉淀，通常有三个信号：重复使用、容易出错、下次还会问。

第二步，采集。

来源要写清楚。网页、PDF、内部文件、客户材料、公开裁判文书，来源不同，后面的使用边界不同。很多知识库的问题，不是答案错，而是来源混在一起，公开资料和内部资料没有隔离。

第三步，加工。

加工不是美化排版，而是让材料变得可处理。文件名统一，页码保留，扫描件转文字，表格字段固定，重复材料去重，重要段落加标签。

第四步，治理。

这里才轮到 Git、脱敏、权限和 .gitignore。

.gitignore 是一个很小的文件，但在法律资料场景里特别重要。它告诉系统：哪些文件永远不要进入版本库。比如客户原始材料、身份证件、未脱敏合同、带内部意见的工作底稿、密钥和配置文件。

第五步，应用。

材料处理完，不代表有用。至少要跑一次真实使用：检索、问答、写作、生成清单、做对比表，任选一种。跑不出结果，就说明前面的结构化还不够。

第六步，发布判断。

这是法律人最不能省的一步。成果更适合自己用、团队共享，还是可对外分享？如果要对外，必须有脱敏版和公开版，不能把内部版换个文件名就发布。

三个最低动作：两次提交、一个忽略表、一个标签

Day 6 作业里，我要求至少 2 次 commit、配置 .gitignore，再打一个 git tag。

不是为了增加难度。

这三个动作刚好对应法律资料治理的三个底线。

两次 commit，解决“过程不可见”的问题。

第一次可以是“导入原始公开资料并记录来源”。第二次可以是“完成脱敏和字段清洗”。这两次提交之间，系统会保留具体差异。以后发现清洗错了，能回看；发现删多了，能恢复。

.gitignore，解决“不该进库的东西进库”的问题。

法律资料最怕顺手。顺手把客户原件拖进文件夹，顺手把带个人信息的表格推到共享仓库，顺手把 API 密钥也放进去。.gitignore 不是万能，但它能给顺手加一道闸。

git tag，解决“哪个版本可交付”的问题。

一个资料包可能持续迭代，但不是每一刻都适合交付。打 tag 的意思是：这个节点可以作为一个稳定版本。比如 v0.1-internal、v0.2-redacted、v1.0-public。名字不用复杂，关键是团队能看懂。

法律资料的麻烦，不在技术上

真实摩擦通常不在命令行。

第一处摩擦是脱敏。

很多人以为把公司名替换成“某公司”就够了。实际项目里，金额、地点、时间、交易结构、岗位名称、文件编号都可能反向定位。脱敏不是替换词，而是重新判断可识别性。

第二处摩擦是来源。

公众号文章、法院案例、客户材料、团队经验，不能放在同一个权威层级里。AI 不会天然知道哪一类更可靠。你不在文件层面标注来源，它就会在回答里混用。

第三处摩擦是版本。

法律判断会变。法规更新、司法解释出台、裁判口径变化，都会让旧材料失效。没有版本记录，团队很难知道某份清单是 2024 年的判断，还是 2026 年已经更新过。

这些问题不用 Git 也能管，但 Git 把它们变成了可检查的动作。

不是什么资料都适合上 GitHub

这里要说边界。

我不主张把所有法律资料都放到 GitHub，更不主张把案件材料拿去公开演示。GitHub Pages 很适合发布可公开的课程资料、法规索引、脱敏模板和方法说明；但客户材料、案件证据、内部法律意见，原则上只适合本地私有仓库或受控的团队环境。

更稳妥的做法，是分三层：

内部版：保留完整工作需要的信息，只在授权范围内流转。

脱敏版：用于团队训练和复盘，去掉可定位信息。

公开版：只保留方法、结构和示例，不保留任何可能指向客户或项目的细节。

法律人的知识资产，价值不在“我有一堆文件”。价值在于这些文件能被追溯、被复用、被验证，也能在该停下的时候停下。

AI 让资料调用变得太容易了。

越容易调用，越要先把账本建好。
