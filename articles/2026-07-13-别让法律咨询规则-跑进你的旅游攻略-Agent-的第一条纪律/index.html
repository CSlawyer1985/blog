别让法律咨询规则，跑进你的旅游攻略——Agent 的第一条纪律

本文目录 一个真实的翻车场景 原理：全局规则 vs 工作空间规则 怎么建一个干净的工作空间 几个容易踩的坑 它能做到什么，做不到什么

用 Agent 的人，第一条纪律不是学写规则，是先建工作空间。

这件事听起来是细枝末节，但用得越久越发现，它决定了你的 AI 三个月后是个清爽好用的搭档，还是一个对所有事情都啰嗦的怪东西。

一个真实的翻车场景

你在 Agent 里写了一条规则：

回答任何法律问题前，先去本地知识库检索、做法规核验、按法律意见书结构输出。

本意是给法律咨询用的。但你不小心把这条规则放在了根目录。

结果下一次，你在同一个产品里让它帮你做个旅游攻略。它上来先跑了一遍法规核验，再按"法律意见书"的结构，给你输出了"旅游攻略法律风险评估意见"。

攻略没出来，倒出来一份意见书。

这就是规则污染。

规则跑错了场景，变成了噪音。还顺手烧掉了一批 token。

原理：全局规则 vs 工作空间规则

Agent 加载规则，是有层级的：

根目录（用户级）：每次对话都加载，不分场景。适合放"我是谁""我的工作偏好"这种跨场景通用的信息。

工作空间（项目级）：只在某个文件夹里对话时才加载。适合放场景专属规则——法律咨询怎么干、合同审核怎么审、讲座怎么准备。

学过编程的人一眼就懂，这就是作用域：全局变量 vs 局部变量。

法律咨询规则是局部变量。你把它定义成全局变量，它就会污染所有场景。做攻略的时候它来插一脚，审合同的时候它也来插一脚，写个朋友圈它还想给你先做一轮法规核验。

不是规则错了，是规则放错了位置。

怎么建一个干净的工作空间

操作不复杂，五步。

第一步，列出你的高频场景。 法律咨询、合同审核、案件分析、讲座准备、内容写作——先想清楚你重复在做的到底是哪几件事。不要凭想象列，翻一下最近一个月的对话记录，哪些任务反复出现，那些就是你的场景。

第二步，给每个场景建一个独立文件夹。 命名要一眼能认出来。法律咨询、合同审核、讲座准备，别起花哨的名字。文件夹就放在你常用的根目录下。

第三步，干活前先选工作空间。 这一步最容易被忽略。打开 Agent 第一件事不是输入问题，是先选今天在哪个文件夹里干活。选错了，后面全错。

第四步，场景专属规则写到工作空间内。 WorkBuddy 会在工作空间里建一个 .workbody/ 隐藏文件夹，Claude Code 用 .claude/，Cursor 用 .cursor/，名字不同，逻辑一样。规则写在这些子文件夹里，不会泄漏到外面。

第五步，只把通用的留给根目录。 "我是律师""我喜欢结论先行""输出用 Markdown"这类跨场景的，放根目录。场景专属的，一律进工作空间。

判断标准很简单：这条规则换一个场景还成立吗？ 成立 → 根目录。不成立 → 工作空间。

几个容易踩的坑

坑一：嫌麻烦，所有规则都堆根目录。

刚开始用 Agent 的人最容易这样。反正根目录写着就能生效，干嘛还建文件夹。三个月后你会发现，你的 AI 变成了一个对所有事情都啰嗦的怪东西——让它写个朋友圈，它也要先做事实梳理和风险分级。这时候你再回头去清理根目录，比一开始就分好类麻烦十倍。

坑二：工作空间建得太细。

有人一听工作空间好，立刻建了"离婚纠纷-财产分割-房产""离婚纠纷-财产分割-股权""离婚纠纷-抚养权"这种四级目录。没必要。工作空间是用来分场景的，不是用来分案由的。二级够用——"法律咨询"下面不要再分，每次咨询一个独立对话就行。

坑三：跨场景规则也塞进工作空间。

"我是律师""我喜欢结论先行"这种身份和偏好信息，应该放根目录。如果你把它塞进"法律咨询"工作空间，下次你进"合同审核"工作空间，AI 就忘了你是律师，开始用通用口吻跟你说话。通用信息要放通用位置。

坑四：以为规则越多越好。

规则越多，互相打架的概率越高。AI 在两条矛盾的规则之间无所适从，输出反而变差。一个工作空间里的核心规则，控制在 5 到 10 条以内最好。多了就该做减法，不是加法。

它能做到什么，做不到什么

建工作空间不能让 AI 变聪明。它能让 AI 在正确的场景里做正确的事，不被无关规则干扰。这听起来不算什么，但实际用起来，差别是数量级的。

一个局限要说清楚：工作空间的隔离不是绝对的。有些产品会"向上查找"——在工作空间里没找到的规则，它会去根目录找。所以根目录放什么、工作空间放什么，要想清楚，别指望靠"放进去就绝对隔离"。

还有一个经验：真正成熟的使用者，工作空间数量会稳定在 5 到 8 个，不会无限膨胀。如果你的工作空间已经建到了 15 个以上，大概率是该合并了。场景分得太碎，本身就是另一种混乱。

第一条纪律就这一句：先建工作空间，再写规则。 顺序反了，后面全是补丁。
