2026 年 7 月 8 日下午两点，我在宁波银行总行讲了一堂 AI Agent 法律场景实践课，时长三小时。讲完打开视频号，刷到腾讯 WorkBuddy 创始人汪晟杰的访谈。看了几分钟我就愣了——他在访谈里说的几乎每一个核心判断，和我刚在课上讲的完全吻合。不是"英雄所见略同"那种模糊的重合，而是概念命名、底层逻辑、甚至比喻方式都一一对应。这不是巧合。

下午的课，我从 Agent 的底层结构讲起。开场就问了一个问题：Chatbot 和 Agent 到底差在哪？

我的回答是：Chatbot 只有嘴，Agent 有手。Chatbot 能回答你，但落不了地——你得自己把答案复制粘贴到文件里。Agent 能直接在你电脑上建文件夹、写文件、改文件、调外部工具，按你设定的规则从头跑到尾。

晚上看汪晟杰的访谈，他的表述是：WorkBuddy 不是聊天工具，是"AI 打工人"。

同一句话，两种说法。

Agent = 模型 + Harness。这是我下午课程的核心公式，也是汪晟杰的产品哲学。

下午我在白板上写了这个等式：Agent = 模型 + Harness。 模型是大脑，Harness 是控制框架——原意是马具，烈马需要缰绳和嚼子，大模型也一样。大模型天生不确定，同样的输入大概率产出不同的输出。这在闲聊场景下没什么，在工作场景下是致命的。Harness 的作用就是控制这种不确定性，让输出稳定、可预期、贴合你的工作场景。

汪晟杰的访谈里，几乎是同一个判断。他说：同一个模型，用不同的控制框架，效果天差地别。WorkBuddy 的底层来自 CodeBuddy——腾讯做了三年的 AI 编程工具——把 AI Coding 的任务执行范式、工具调用能力、上下文管理迁移到办公场景。

一个是从使用者端讲"为什么 Harness 比模型重要"，一个是从产品端讲"为什么我们要把 AI Coding 的基建复用到办公场景"。同一个逻辑的两面。

关于"模型不会拉齐"——我们给出了同一个判断，用了同一个比喻。

下午课上有人问：现在模型这么多，该选哪个？

我的回答是：头部模型在法律场景里都够用了。95 分和 100 分的差距，远没有"你怎么用"的差距大。我更建议你把精力放在建规则上，而不是追模型。

汪晟杰的访谈里有一个更彻底的表述：模型一定不会拉齐。他的理由是——大模型不是静态参数，而是训练数据、用户反馈、应用场景、后训练体系共同塑造的动态系统。他用了一个比喻：训练一只小狗听话，需要持续反馈和奖励，但把这智商迁移到另一只狗身上，很难。

他甚至点出了一个我课上没展开讲的逻辑：国内大模型厂商都在做 CLI 和 Agent 产品——Claude Code、Codex、CodeBuddy、WorkBuddy、Trae——因为这些产品不只是用户工具，更是模型进化的数据入口。谁拥有更高频、更真实的用户任务场景，谁就拥有更好的模型进化土壤。

这个判断比我课上的更锋利。但我课上的结论和他殊途同归：不要追模型，要建自己的 harness。

工作区规则不是写出来的，是长出来的。这是下午课的核心方法论，也是 WorkBuddy 的产品设计前提。

下午课上我花了最长的时间讲一件事：怎么从零构建一个工作区。

我在现场演示了一个完整流程——新建"法律咨询"文件夹，用 WorkBuddy 打开，在对话框里描述需求：

"每次法律咨询用时间+关键词新建子文件夹。先理解问题，拆解，检索本地知识库，按权威等级规则做研究。然后调用北大法宝 MCP 核验所有法条的准确性。所有来源标注标签。默认输出 markdown 文件。"

三分钟，WorkBuddy 自动生成了完整的规则文件。

然后我现场测试了一个问题——"担保保证期间的法律规定"。Agent 按规则新建了文件夹，拆分了子问题，检索了本地知识库，调用了北大法宝，输出了结构化解答。全程不需要我写一行代码。

但重点不在"三分钟建工作区"。重点在接下来的步骤：发现问题，迭代规则。

第一次测试跑完，我发现两个问题：原始问题没有存档，"结论先行"的表述不对我的口味。就在对话框里说了一句——"原始问题必须存档。'结论先行'改成'结论'。"下一次测试，全改过来了。

我在课上反复说：规则不是一蹴而就的，是一步一步生长出来的。每次纠正一次，规则就厚一层。一个月后，你就会有一个完全适配你工作习惯的规则体系。 这就是把隐性知识变成显性规则的过程——你大脑里的专业判断，通过一次次纠错，蒸馏成了 Agent 的行为规则。

汪晟杰的访谈里没有讲这么细的操作，但他产品设计的底层假设和我的方法论完全一致。他说 WorkBuddy 做 PPT 做得好，不是因为模型会做 PPT，而是因为它在用户的电脑里积累了资料、风格、偏好——"我的电脑换到别人那边去，它就不懂了。"这就是工作区记忆。他说 Vibe Coding 的核心是"创意为导向，你的想法能跟 AI 说明白，就能不断对话、记录过程、形成产物"。这不就是"每次纠正一次，规则就厚一层"？

一个观点，两个独立来源，同日发表。

这件事最有意思的地方，不是汪晟杰的观点和我一致——而是我们是从完全不同的起点，走到了完全相同的结论。

我从一个律师的角度出发。两年前开始用 Claude Code，从命令行学起，踩了无数坑。慢慢摸索出一套方法论：按场景建工作区，写 CLAUDE.md，拆 Skill，积累 memory，每次纠错就沉淀一条规则。这套方法不是看论文看出来的，是用了两年 Agent、做了几百次法律咨询、审了无数合同之后，自然形成的。

汪晟杰从一个产品副总裁的角度出发。他在腾讯做了三年 CodeBuddy，看着 AI Coding 的基建越来越厚，然后决定把这套能力外溢到办公场景。他的方法论不是写文章写出来的，是产品迭代、用户反馈、工程实践逼出来的。

一个法律人，一个产品人。一个从使用者端往上拱，一个从产品端往下探。在同一天，说了同一套话。 这意味着什么？意味着这不是某个人的偏好或运气，而是 Agent 产品的底层规律——不管你是谁、从哪个方向进入，只要你深入到这个层面，你都会得出同样的结论。

对法律行业意味着什么

下午的课，宁波银行的同事们提了很多实务问题：长周期案件怎么保持状态？幻觉怎么控制？保密怎么处理？Agent 权限太大怎么办？

我的回答归结起来是一个方向：法律人要从"亲手做每一步"转向"设计流程、把关依据、承担判断"。

汪晟杰的访谈给了一个产品端的佐证。他说 WorkBuddy 的高频场景之一是"把复杂文字变成 HTML 报告和交互小应用"。他说未来的 Agent 不是聊天框，而是一个持续运行的工作体——电脑上执行任务，手机上查看进度，云端保存状态。

对法律行业来说，这意味着：真正有价值的法律 AI 产品，不是"法律 ChatGPT"，而是一组能持续理解案件背景、客户偏好、律师风格和交付标准的 Agent 系统。它们的关键不是模型能力，而是是否接入了资料库、是否理解个人风格、是否能长期记忆、是否能嵌入真实办案流程。

这也是我在课上反复讲的最后一句话：你的规则，就是你的核心竞争力。 工具可以换，模型可以换，但你沉淀的规则和数据是你的个人资产。汪晟杰的产品可以做得很漂亮，但它只能给你通用预设。只有你自己，在使用中发现问题、纠错、迭代，才能养出一个真正贴合你工作场景的 Agent。

模型不是终局，Harness 才是。

工具不是终局，规则才是。

一次生成不是终局，长期记忆和反馈闭环才是护城河。
